Changelog
=========

Development
-----------

* `comparison_groups` guards. `savings/model_correction`: a zero comparison-group model magnitude (percent / absolute-percent difference-in-differences) is guarded to scale 0 in both the correction and its variance, so it contributes no correction and stays finite instead of dividing to inf. `individual_meter_matching`: raise `DistanceMatchingError` on an empty comparison pool, and when `max_distance_threshold` filters out every candidate match. `cg_clustering/bounds`: remove the dead always-false `linear` lower-bound branch.
* `clustering` / `comparison_groups` correctness tests (no behavior change). Clustering: extend the shared algorithm harness with `kmedians` / `bisecting_kmedians` and assert ground-truth recovery (ARI) rather than a majority heuristic; parametrize the degenerate-data fallback over the honest-degeneracy family with an explicit spectral forced-k contract; new per-algorithm suites (`k_medians`, `bisect_k_medians`, `dbscan`, `birch`, the vendored `sklearn_bisect_k_means` fork delta, `affinity`, `spectral_divisive`, `parallel_analysis`, `clustering_settings`); validity-index monotonicity-in-separation and hand-computed / sklearn-reference values; `has_cluster_structure` grouped-agreement branch logic; DWT roundtrip; voter-discriminability, `prepare_labels`, k-penalty, and the `cluster_features` row-order contract. Comparison groups: IMM matches equal brute-force nearest neighbours; direct `t_and_ks_test`; IMM / HiGHS settings validators; base-class `get_loadshapes`-before-fit contract; a deliberately-mismatched population that fails stratified equivalence; and a `model_correction` regression snapshot across hourly / daily / billing × ODID / PCTDID / ABSPCTDID, pinned on real ComStock-derived inputs (committed `.npz` fixtures generated by `tests/comparison_groups/savings/generate_correction_fixtures.py`).
* Test layout: reorganize the test tree to mirror the source package (no behavior change). Move the top-level `test_caltrack_*` / `test_segmentation` / `test_derivatives` into `tests/eemeter/hourly_caltrack/`; `test_features` / `test_transform` / `test_exceptions` / `test_warnings` into `tests/eemeter/common/`; the `common/stats` transform/loss/outlier tests (and the misplaced `test_adaptive_loss`) into `tests/common/stats/`; and the billing tests out of `tests/eemeter/daily_model/` into `tests/eemeter/billing_model/`. Snapshots travel with their tests. Make `tests/eemeter/` a package so its subpackages namespace under `eemeter.*` (avoids a `common` basename collision with `tests/common/`). In `test_derivatives.py`, the session fixtures now `.copy()` the shared `comstock` data before constructing `Daily`/`Billing` data objects, so the new alphabetical collection order can't leak a tz-converted frame into the design-matrix tests; the in-place mutation in the data classes themselves is tracked separately. Drop `tests/test_version.py` (a bare `__version__` major-version assertion — no behavioral coverage).
* CI: the `update-win-snapshots` workflow is now `workflow_dispatch`-only (was per-PR, where auto-regenerating would have blessed Windows output rather than pinning a baseline). Fix its artifact glob to `*_win*.json` (the prior `**/*_win.json` never matched the `[regression_win]` snapshot names, so it committed nothing) and repoint the `test_derivatives` snapshots to `tests/eemeter/hourly_caltrack/__snapshots__/`.
* Test/coverage infrastructure. Collect the previously-orphaned `tests/common/metrics.py` (renamed to `test_metrics.py` — its `test_acf` never ran because the filename lacked the `test_` prefix) and correct its stale expectations to `acf`'s actual per-method output. Add a `tox -e coverage` environment that runs serially under `NUMBA_DISABLE_JIT=1` so `@numba.jit` functions are visible to the coverage tracer (they execute as native code and report as uncovered otherwise); add `[tool.coverage.report]` `exclude_lines` and omit the `comparison_groups/savings/dev/` work-in-progress scratch. An `xfail` records that `acf(ac_type=STATIONARY_STATS_FFT)` omits zero-padding and so returns the circular (not linear) autocorrelation.
* `common/hourly_interpolation`: `_autocorr_fcn` computes the autocorrelation via FFT above 16 lags and the masked loop at or below, returning identical values (missing data is zero-filled after centering, so it contributes 0 to each lag product). Remove the unused `_autocorr_fcn2` / `_autocorr_fcn3` reference variants; raise on an unknown backup interpolation method. Add direct characterization tests pinning both branches to the original output.
* Bug fix (`clustering`): degenerate-data handling across the selection and algorithm layers.
  - `ClusteringResult`: `.k` / `.metrics` / `.labels` raised `IndexError` when every candidate labeling was rejected by `prepare_labels` (empty `_labels_store`, e.g. all collapsed below `n_cluster_lower`). They now raise a clear `ValueError` naming `n_cluster_lower` when `k=1` is disallowed, and return the single cluster when it is allowed.
  - `ClusteringResult`: when valid candidates exist but the council produces no scored winner, the fallback returns the labeling with the fewest clusters (`>= n_cluster_lower`) instead of the insertion-order-first one, and never a count below the lower bound.
  - `ClusteringResult`: a single specified k (one distinct `k`) bypasses the gap gate and cross-k council entirely, returning the within-k best with full confidence.
  - `SingleKMetrics`: add `unique_valid_labels` / `valid_label_count` (outlier label `-1` excluded); `ClusteringResult.k` uses the valid count.
  - `bisect_k_medians`: bound the per-node split retries so unsplittable data (e.g. identical points) no longer loops forever — after the retry budget the node is accepted as a terminal cluster.
  - `spectral`: the sparse self-tuning path falls back to a dense `eigh` of the smallest eigenpairs when ARPACK fails to converge (previously re-raised).

opendsm-1.2.5 (2025-07-24)
--------------------------

* Spectral clustering's `assign_labels` is now configurable via a new `SpectralAssignLabels` enum (`kmeans`, `discretize`, `cluster_qr`) instead of being hardcoded to `cluster_qr`; note that `discretize` and `cluster_qr` are not guaranteed deterministic under a fixed seed.
* Added new model-fit metrics to `BaselineMetrics`: Nash-Sutcliffe Efficiency (`nse`), Normalized NSE (`nnse`), Kling-Gupta Efficiency (`kge`), and accuracy-within-tolerance metrics `a10`/`a20`/`a30` (fraction of points with relative residual within 10/20/30%).
* Renamed the Confidence Index to Performance Index: `ci`/`ci_rating` are now `pi`/`pi_rating` (same `pearson_r * wi` formula and rating thresholds).
* Refactored several `BaselineMetrics` computations (WAPE, sWAPE, Willmott Index `wi`, Legates-McCabe, `explained_variance_score`) to reuse cached `mae`/`sse`/mean/variance values; behavior is unchanged except these no longer apply per-element finite masking before summation.

opendsm-1.2.4 (2025-07-18)
--------------------------

* Fixed `BaselineMetrics.ci_rating` boundary bug: thresholds used overlapping/gapped bounds (e.g. `ci > 0.85` excluded exactly 0.85, and ranges like `0.76 <= ci <= 0.85` left values such as 0.85-0.86 and 0.75-0.76 unclassified). Rating bands now use clean half-open intervals, so 0.85 maps to `excellent` and every CI value falls into exactly one band.
* Added autocorrelation-normalized error metrics to `BaselineMetrics`: `rmse_autocorr` (RMSE computed against the effective sample size `n_prime`), plus its normalized variants `cvrmse_autocorr` and `pnrmse_autocorr`.

opendsm-1.2.3 (2025-07-18)
--------------------------

eemeter-0.4.14-alpha (2017-01-04)
---------------------------------

* Replaced the weather cache backend `SqliteJSONStore` with `SqlJSONStore`, built on SQLAlchemy: the cache is now configured by a database URL (e.g. `sqlite:///...`) instead of a directory path, allowing non-SQLite backends.
* Cache location is now read from the `EEMETER_WEATHER_CACHE_URL` environment variable (replacing the former `EEMETER_WEATHER_CACHE_DIRECTORY`); it still defaults to a SQLite file under `~/.eemeter/cache`.
* Added `SQLAlchemy` as a runtime dependency.

eemeter-0.4.13-beta (2016-12-13)
--------------------------------

* First tagged release of the Open Energy Efficiency Meter (`eemeter`); subsumes the entire 0.0.x–0.4.x development line (none of which were git-tagged) into the trace-centric architecture.
* Trace-centric data model under `eemeter.structures`: `EnergyTrace`/`EnergyTraceSet`, `ModelingPeriod`/`ModelingPeriodSet`, `Intervention`, `Site`, and `Project` replace the older consumption/project objects.
* Energy-efficiency meter pipeline in `eemeter.ee`: `EnergyEfficiencyMeter` for trace-centric metering, an aggregation class for portfolio-level rollups (with serialized status and outputs), and derivative-pair computation (savings, gross predicted, annualized) with upper/lower confidence bounds.
* Modeling layer (`eemeter.modeling`): `SeasonalElasticNetCVModel` and a billing-period model built on `ElasticNetCV`, with bootstrap-estimated empirical errors; model fit output always returned, including start/end dates and tracebacks on failure.
* Weather subsystem (`eemeter.weather`) rewritten around a pandas-index-based API: `ISDWeatherSource` (hourly + daily temperatures), `GSOD` and `TMY3`/`CZ2010` normal sources, lazy loading of station/TMY3 indices and FTP connections, UTC/timezone-aware handling, and a swappable caching layer (no ORM).
* Location resolution via in-memory station/climate-zone/ZIP indices using haversine nearest-station lookup; raises clear errors on bad ZIP/station and handles null location fields.
* I/O (`eemeter.io`): ESPI/Green Button XML parsers (electricity + natural gas, forward/reverse summing, `has_solar` and net-metering support) plus serializers for traces, meter input/output, and aggregation input.
* Input/output serialization for meters and aggregations: takes serialized records via `to_records`/`from_records`, requires derivative labels, and emits serialized modeled energy traces and demand fixtures.
* Python 2/3 compatibility throughout (via `six`), structured logging across meter/model/weather, and bundled station/ZIP/climate-zone resource data.
* Documentation overhaul: trace-centric tutorials and Jupyter notebooks for `eemeter` and the datastore, glossary, methods overview, and processor/model docs.
